# Proposal: The Lab As A Living Environment For Agent Evolution

> Status: Draft
> Created: 2026-03-07
> Decision: Decide whether continuous evolution measurement is important enough to justify a separate subsystem
> Scope: A persistent, observable lab environment that measures long-term agent improvement under controlled entropy
> Next Step: If the idea is accepted, write an implementation plan for Phase A (`Greenhouse`) only
> Related: [2026-03-07-e2e-testing.md](2026-03-07-e2e-testing.md), [../design/evolution.md](../design/evolution.md)

## Summary

This proposal reframes "evolution testing" as a continuous lab instead of a periodic benchmark batch.
The lab would give Sprout a living codebase, controlled problem injection, and a health observatory that plots change over time.
It is intentionally larger than a test suite and should only move forward if long-horizon measurement is a real project priority.
If adopted, it should start with the smallest independently valuable slice: the `Greenhouse` codebase.

## Problem

The original e2e-testing design treats evolution measurement as a batch job: "run 30 problems, compare batch 1 vs batch 3." But this misses the point. Evolution is not a test you run — it's a property you **observe over time in a living system**.

The right analogy is not a unit test suite. It's a **laboratory**: a controlled environment where the organism lives, acts, and changes — and where the experimenter can observe everything, inject perturbations, and measure the result.

## What Is the Lab?

A **persistent, observable, self-replenishing codebase environment** that the agent operates on continuously. The lab has three properties that static test fixtures lack:

1. **Statefulness** — the agent's past work persists. Solving problem A changes the environment for problem B. Cumulative effects are visible.
2. **Entropy injection** — the environment doesn't just sit there. New problems appear over time (controlled mutations, simulated code rot, new feature requests). The agent must swim against a current.
3. **Continuous measurement** — health metrics are always being computed, not just at test boundaries. You can plot a single curve of "codebase health over time" and see evolution as a shape, not a number.

The evolution signal becomes concrete:

```
                   Codebase Health
                   ^
                   |     ╱──╲        ╱────────
                   |    ╱    ╲──╲   ╱
                   |   ╱        ╲──╱
                   |  ╱
                   | ╱
                   |╱
                   └──────────────────────────> Time
                     ▼m  ▲f  ▼m ▼m ▲f ▲f
                     m = mutation injected
                     f = agent fix landed
```

A healthy agent shows: (a) recovery after each mutation, (b) shorter recovery times over weeks, (c) health baseline trending upward despite ongoing entropy.

### Component 1: Living Codebase

Not a test fixture. A **project** — with the structural complexity of a real codebase — that the agent treats as its working environment.

**Requirements:**

- Multiple modules with inter-dependencies (changing module A can break module B)
- A working test suite with measurable coverage
- Linting rules and measurable lint score
- Documentation with measurable completeness
- A git history (the agent operates through normal git workflow)
- Enough complexity that "understanding the codebase" is a real challenge (~2000-5000 LOC)

**Options for the initial Living Codebase:**

| Option | Pros | Cons |
|--------|------|------|
| **Synthetic project** (purpose-built) | Full control over ground truth, can design complexity precisely | Doesn't capture real-world messiness |
| **Real OSS fork** (e.g., a small Python lib) | Realistic | Hard to measure ground truth, uncontrolled complexity |
| **Sprout itself** (self-referential) | Maximum alignment, agent improves its own lab | Dangerous, can't run uncontrolled experiments |

**Recommendation: Synthetic project, designed for observability.** A purpose-built Python project (e.g., a small web API with auth, storage, background jobs, CLI) where every module's purpose, inter-dependency, and test coverage is known by construction. The project should feel real — not toy code — but every aspect is measurable.

**Candidate: "Greenhouse"** — a synthetic Python project that serves as Sprout's lab environment. A small but structurally complete application:

```
greenhouse/
├── src/
│   ├── api/              # HTTP endpoints (3-4 routes)
│   ├── auth/             # Authentication (token-based)
│   ├── storage/          # Database layer (SQLite)
│   ├── jobs/             # Background task runner
│   ├── notifications/    # Email/webhook sender
│   └── cli/              # Command-line interface
├── tests/
│   ├── test_api.py
│   ├── test_auth.py
│   ├── test_storage.py
│   └── ...
├── docs/
│   └── ...
├── pyproject.toml
└── .github/
    └── workflows/
```

~3000 LOC. Each module has known coupling points. Test coverage is measurable. Lint score is measurable. The entire project is designed so that the Health Observatory can produce precise readings.

### Component 2: Problem Generator

The entropy engine. Without it, the agent solves all problems and the lab goes silent. The Problem Generator keeps the environment dynamic — like a continuous supply of challenges at a controllable rate.

**Mutator Types:**

| Mutator | What it does | Difficulty | Detection |
|---------|-------------|------------|-----------|
| `SyntaxBreaker` | Introduces a syntax error in a random file | 1 (trivial) | lint, import |
| `ImportRemover` | Deletes a used import statement | 2 (easy) | test failure, import error |
| `LogicInverter` | Flips a boolean condition | 3 (medium) | test failure (if covered) |
| `TodoPlanter` | Adds a realistic TODO comment with context | 2 (easy) | todo scan |
| `TestDeleter` | Removes a test, reducing coverage | 2 (easy) | coverage drop |
| `DeadCodeAdder` | Adds an unused function/import | 1 (trivial) | lint |
| `DependencyBreaker` | Changes a function signature, breaking callers | 4 (hard) | test failure, multiple files |
| `SecurityHolePuncher` | Introduces an input validation gap | 4 (hard) | security scan, subtle |
| `DocDrifter` | Makes a docstring inaccurate (code changed but doc didn't) | 3 (medium) | doc/code consistency |
| `PerformanceDegrader` | Adds an O(n^2) loop where O(n) existed | 3 (medium) | benchmark regression |

**Mutator Protocol:**

```python
@dataclass
class Mutation:
    id: str                     # unique identifier
    mutator: str                # "SyntaxBreaker", "LogicInverter", etc.
    target_file: str            # which file was mutated
    difficulty: int             # 1-5
    injected_at: str            # ISO timestamp
    description: str            # human-readable description of what was changed
    ground_truth_fix: str       # the diff that would perfectly reverse this mutation
    detected_at: str | None     # when agent discovered it (None = undetected)
    fixed_at: str | None        # when agent's fix landed (None = unfixed)
    agent_fix_correct: bool | None  # was the agent's fix equivalent to ground truth?

class Mutator(Protocol):
    def can_apply(self, codebase: Path) -> bool:
        """Check if this mutator can be applied to the current codebase state."""

    def apply(self, codebase: Path) -> Mutation:
        """Apply mutation, return metadata including ground truth fix."""

    def verify_fix(self, codebase: Path, agent_diff: str, ground_truth: str) -> bool:
        """Check if the agent's fix is functionally equivalent to ground truth."""
```

**Scheduling:**

The Problem Generator runs on a schedule independent of the agent:

```python
@dataclass
class GeneratorConfig:
    mutations_per_day: int          # rate of entropy injection
    difficulty_distribution: dict   # {1: 0.3, 2: 0.3, 3: 0.2, 4: 0.15, 5: 0.05}
    mutator_weights: dict           # relative frequency of each mutator type
    max_active_mutations: int       # don't overwhelm — cap unresolved mutations
    cooldown_after_mutation: int    # seconds between mutations (prevent clustering)
```

**Key design principle:** The generator's rate is **tunable**. During early experiments, inject slowly (1-2/day) and mostly easy (difficulty 1-2). As the agent proves capable, increase rate and difficulty. This is the "curriculum" — the lab adapts to the agent's level.

### Component 3: Health Observatory

Continuous measurement of both the codebase and the agent. The observatory produces **time series**, not point-in-time snapshots.

**Codebase Health Metrics (computed periodically, e.g., after every agent commit and every mutation):**

```python
@dataclass
class CodebaseHealthSnapshot:
    timestamp: str
    trigger: str                    # "agent_commit" | "mutation_injected" | "scheduled"

    # Correctness
    test_pass_rate: float           # % of tests passing
    test_count: int                 # total test count
    syntax_errors: int              # files that don't compile

    # Quality
    lint_score: float               # 0.0-1.0 (lint violations / LOC)
    test_coverage: float            # line coverage %
    cyclomatic_complexity_avg: float
    cyclomatic_complexity_max: float
    dead_code_count: int            # unused imports, unreachable functions

    # Maintenance
    todo_count: int
    doc_coverage: float             # % of public functions with docstrings
    type_annotation_coverage: float

    # Security
    known_vulnerabilities: int      # from security scanner
    input_validation_gaps: int

    # Size
    total_loc: int
    file_count: int
```

**Agent Performance Metrics (computed per task):**

```python
@dataclass
class AgentTaskMetrics:
    task_id: str
    mutation_id: str | None         # which mutation was this task addressing? (None if organic)
    timestamp: str

    # Effectiveness
    outcome: str                    # "completed" | "failed" | "needs_human"
    fix_correct: bool | None        # compared against ground truth (if mutation-originated)
    replan_rounds: int              # 0 = first attempt success

    # Efficiency
    tokens_used: int
    wall_clock_seconds: float
    plan_steps: int
    executed_steps: int

    # Learning
    experiences_recalled: int
    experiences_relevant: int       # post-hoc judgment
    new_experiences_created: int
    experiences_reinforced: int     # existing knowledge confirmed
```

**Derived Evolution Signals (computed over sliding windows):**

```python
@dataclass
class EvolutionSignals:
    window: str                     # "7d" | "30d" | "all"

    # Recovery dynamics
    mean_time_to_detect: float      # avg time from mutation injection to discovery
    mean_time_to_fix: float         # avg time from discovery to fix landed
    recovery_rate: float            # % of mutations resolved within window

    # Efficiency trajectory
    tokens_per_fix_trend: float     # slope of tokens-per-successful-fix over time
    first_attempt_rate_trend: float # slope of first-attempt success rate

    # Learning quality
    recall_precision_trend: float   # slope of experience recall precision
    duplication_rate: float         # % of new experiences that duplicate existing

    # Difficulty frontier
    max_difficulty_solved: int      # hardest mutation the agent has resolved
    difficulty_success_curve: dict  # {difficulty: success_rate}

    # Net health impact
    health_delta: float             # codebase health now vs window start
    entropy_injected: float         # total mutation difficulty injected
    entropy_resolved: float         # total mutation difficulty resolved
    net_entropy: float              # injected - resolved (negative = agent winning)
```

The **single most important number**: `net_entropy` over a 30-day window. If it's negative and becoming more negative, the agent is evolving. If it's positive, the agent is falling behind the problem generator — time to investigate.

### Component 4: Experiment Controller

The scientific method for agent development. Without controlled experiments, you can't attribute changes in agent behavior to specific code changes.

**Core Operations:**

```python
class ExperimentController:

    def fork(self, name: str) -> LabInstance:
        """Create an independent copy of the current lab state.

        Use case: A/B testing. Fork, modify agent code in one branch,
        run both for a week, compare evolution signals.
        """

    def checkpoint(self, name: str) -> Checkpoint:
        """Save complete lab state (codebase + agent state + metrics history).

        Use case: Before a risky agent change, checkpoint. If the change
        degrades performance, restore and try a different approach.
        """

    def restore(self, checkpoint: Checkpoint):
        """Restore lab to a previous checkpoint."""

    def inject(self, mutation: Mutation):
        """Manually inject a specific mutation.

        Use case: Targeted testing. "I improved the planner for refactoring
        tasks. Let me inject 5 DependencyBreaker mutations and see if
        recovery time improves."
        """

    def set_rate(self, mutations_per_day: int):
        """Adjust problem generation rate.

        Use case: Crank up entropy to stress-test. Or lower it during
        a focused experiment on learning quality.
        """

    def compare(self, a: LabInstance, b: LabInstance) -> ComparisonReport:
        """Compare evolution signals between two lab instances.

        Use case: "Agent v2.1 vs v2.0 — which recovers faster from
        DependencyBreaker mutations?"
        """
```

**A/B Testing Protocol:**

```
1. Checkpoint current lab state
2. Fork into lab-A (control) and lab-B (experiment)
3. Apply agent code change to lab-B only
4. Run both for N cycles with identical mutation schedules
5. Compare evolution signals
6. If lab-B wins: merge agent change, continue on lab-B state
7. If lab-A wins: discard change, continue on lab-A state
```

This is how you develop the agent scientifically. Not "does this code change pass tests?" but "does this code change make the agent evolve faster?"

## The Evolution Equation

The lab reduces "is the agent evolving?" to a measurable signal:

```
Evolution Rate = d(entropy_resolved) / d(time)
                 ──────────────────────────────
                 d(entropy_injected) / d(time)
```

- **Ratio > 1**: Agent resolves problems faster than they appear. Codebase health improves.
- **Ratio = 1**: Agent treads water. Codebase health stable.
- **Ratio < 1**: Agent falls behind. Codebase health degrades.
- **Ratio increasing over time**: Agent is evolving. **This is the signal we want.**

For a more nuanced view, decompose by difficulty:

```
Evolution Rate(d) = resolve_rate(difficulty=d) / inject_rate(difficulty=d)
```

An agent that evolves should show:
- Evolution Rate(easy) ≈ 1.0 early, stays high (handles easy stuff from day 1)
- Evolution Rate(medium) starts < 1.0, grows toward 1.0 over weeks
- Evolution Rate(hard) starts near 0, slowly increases as agent accumulates experience

This gives you a **difficulty frontier** — the hardest problems the agent can reliably handle — and you can watch that frontier expand over time.

## Relationship to Existing Testing

The lab does NOT replace Layers 1-3 from [2026-03-07-e2e-testing.md](2026-03-07-e2e-testing.md):

| Concern | Where it's tested |
|---------|-------------------|
| Safety invariants (constitution, force push, etc.) | Layer 1 — per-commit, mock LLM, deterministic |
| Pipeline correctness (plan→execute→verify→ship) | Layer 2 — per-PR, recorded LLM |
| Decision quality (plan quality, PR quality) | Layer 3 — per-release, LLM-as-judge |
| **Agent evolution over time** | **The Lab — continuous, live, observable** |

Layers 1-3 are CI gates. The lab is a **continuous observatory**. They serve different purposes and both are necessary.

## Implementation Phases

### Phase A: Greenhouse Project

Build the synthetic codebase. This is the substrate everything else depends on.

- Design module structure (~6 modules, ~3000 LOC)
- Write the actual code (it needs to work — run, pass tests, be lintable)
- Document every module's purpose and inter-dependencies
- Establish baseline health measurements
- Store in `lab/greenhouse/` (or a separate repo)

### Phase B: Health Observatory

The measurement layer. Useless without Phase A, but doesn't require the agent to work.

- `CodebaseHealthSnapshot` computation (coverage, lint, complexity, tests)
- Time series storage (SQLite — consistent with project conventions)
- Basic visualization (CLI table or simple plot output)
- Triggered by git hooks or scheduled polling

### Phase C: Mutator Framework

The entropy engine. Requires Phase A (a codebase to mutate) and Phase B (to measure the effect).

- Mutator protocol and 3-4 initial mutators (SyntaxBreaker, ImportRemover, TodoPlanter, TestDeleter)
- Mutation metadata tracking (injected, detected, fixed)
- Ground truth fix generation and verification
- Scheduling with configurable rate

### Phase D: Agent Integration

Connect the real Sprout agent to the lab. Requires the agent to be functional (current V2 is sufficient).

- Configure agent to use Greenhouse as its workspace
- Map agent tasks to mutations (which mutation triggered this task?)
- Compute AgentTaskMetrics per task
- Compute EvolutionSignals over sliding windows

### Phase E: Experiment Controller

Scientific experimentation. Requires all previous phases.

- Checkpoint/restore (git snapshot + SQLite dump + agent state files)
- Fork (copy lab directory, run independently)
- Compare (diff evolution signals between instances)
- A/B testing protocol automation

## Dependencies and Constraints

- **Phase A has no dependencies** — can start immediately, doesn't require agent changes
- **Phases B-C depend only on Phase A** — they measure and perturb the codebase, independent of the agent
- **Phase D depends on a working agent** — current V2 is sufficient, but Phase 1 Memory upgrade would make evolution measurements more meaningful
- **The lab uses SQLite for its own storage** — consistent with project conventions, no new infrastructure
- **The lab is a separate subsystem** — it observes the agent but does not modify agent code. The agent doesn't know it's in a lab. It just sees a workspace with issues to fix.

## What This Enables

Once the lab is running:

1. **"Is the agent getting better?" has a quantitative answer.** Plot the evolution rate curve. If it's going up, ship. If not, investigate.

2. **Agent development becomes empirical.** Change a prompt → run in the lab → compare evolution rate before and after. No guessing.

3. **Difficulty curriculum is automatic.** Start with easy mutations. As evolution rate for easy stabilizes near 1.0, increase difficulty. The lab self-calibrates to always challenge the agent at its frontier.

4. **Regression detection is continuous.** If a code change degrades the agent, the evolution rate drops. You see it in real-time, not after a monthly batch test.

5. **Phase 1 Memory upgrade has a clear success metric.** Before: recall precision trend is flat. After: recall precision trend is positive. Measured in the lab, not guessed from code review.

6. **Multi-agent comparison becomes trivial.** Fork the lab, run agent-A and agent-B with identical mutation schedules, compare curves. Which one evolves faster?

## Why Now

- The project mission explicitly cares about compounding intelligence over time, so a static test suite is only a partial fit
- Designing the lab now clarifies the boundary between ordinary CI and long-horizon evaluation
- The proposal can stay dormant until the team is ready, but the conceptual split should be explicit

## Risks and Open Questions

- This is a subsystem, not a small testing feature; operational cost and scope are substantial
- A synthetic codebase may be measurable but insufficiently realistic
- If the current runtime still lacks solid Layers 1-3 testing, the lab may be premature
- It is not yet decided whether the lab should live in this repository or as a sibling repo

## Exit Criteria

- Approved: Phase A (`Greenhouse`) gets its own implementation plan in `docs/plans/`
- Rejected: the team decides periodic benchmarks are enough and this document is marked `Rejected`
- Superseded: another proposal or design provides a better long-horizon evaluation model
